## O que é este projeto

Aplicação **desktop Electron** para empresas brasileiras baixarem automaticamente XMLs de **Notas Fiscais de Serviço (NFS-e)** do Portal Nacional do Governo Federal (nfse.gov.br).

Autenticação via **mTLS com Certificado Digital A1 (.pfx)** ou **usuário/senha**. Os XMLs são salvos **em disco local** na máquina que executou a extração; os metadados (chave de acesso, valor, data) ficam no **Supabase (PostgreSQL)**.

## Fluxo Principal (Extração RPA via Scraper)

```
1. Usuário seleciona .pfx + senha  (ou usuário/senha)
2. POST /scraper/validate-cert → valida cert, identifica empresa pelo CNPJ
3. Usuário clica "Iniciar Extração"
4. POST /scraper/fetch-gov:
   a. nfseScraperService autentica: GET /EmissorNacional/Certificado (mTLS TLSv1.2)
   b. 302 → /Dashboard → extrai JWT (window.sessionStorage)
   c. GET /EmissorNacional/Notas/Recebidas?dataInicio=DD/MM/YYYY&dataFim=DD/MM/YYYY
      Authorization: Bearer {JWT}
   d. Parse HTML/JSON com Cheerio
   e. Upsert metadados → Supabase tabela `nfs`
   f. Download XMLs → {output_path}/recebidas/ ou {output_path}/emitidas/
      (padrão: ~/Documents/notas_processadas/)
```

## Fluxo Alternativo (Sync via API gov.br)

```
POST /companies/:id/sync
→ nfseService.syncNfse(companyId)
→ Baixa certificado do Supabase Storage
→ Autentica com mTLS na API REST do gov.br
→ Para cada nota encontrada:
   - Decomprime XML (base64 + gzip)
   - Salva XML em disco: {output_path}/recebidas/{chave}.xml
   - Upsert metadados no Supabase (xml_url = "local_extract/recebidas/{chave}.xml")
```

## Tabelas do Banco (Supabase)

| Tabela        | Principais Colunas                                                        |
|---------------|---------------------------------------------------------------------------|
| `companies`   | id, cnpj, name, certificate_url, certificate_password, login_user, login_password, certificate_local_name |
| `nfs`         | id, company_id, access_key, issue_date, amount, xml_url, status           |
| `sync_logs`   | id, company_id, searched_at, docs_found, status, error_message            |
| `users`       | id, name, access_key, role, plan_type, is_active, expires_at              |
| `app_settings`| id, key, value — ex: `output_path` define onde os XMLs são salvos         |

### Campo `xml_url` (tabela `nfs`)

Caminho relativo para resolver o XML no disco local:
```
local_extract/recebidas/CHAVE.xml
local_extract/emitidas/CHAVE.xml
```
Caminho absoluto resolvido pelo backend: `{output_path}/recebidas/CHAVE.xml`

> XMLs existem **somente na máquina que executou a extração**. O ZIP só funciona se o backend rodar na mesma máquina.

## Convenções Importantes

- **Período máximo de consulta**: 30 dias (limite do portal NFSe Gov.br)
- **TLS**: forçar `maxVersion: 'TLSv1.2'` — IIS 10.0 do gov.br tem bugs com client cert em TLS 1.3
- **Senhas de certificado**: salvas em texto plano no banco (MVP)
- **`SUPABASE_ANON_KEY`** no backend na verdade é a **service_role key** (nome enganoso por legado)
- **Supabase Storage**: usado apenas para certificados (.pfx). XMLs vão para disco local
- **`output_path`**: configurável via `app_settings` no Supabase. Padrão: `~/Documents/notas_processadas`
- **Documentação**: atualizar o doc correspondente em `docs/` antes de cada commit/push
- **Tabela `nfs`**: foi renomeada de `nfs_docs` — migrations SQL registram essa mudança
